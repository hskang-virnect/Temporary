HSK 10자리 품목분류 실험 코드 구조

세 가지 예측 방식이 같은 데이터 분할, 같은 평가 지표, 같은 결과 저장 구조를 공유하도록 구성된 작은 연구용 코드베이스입니다.

실행 흐름

원천 JSONL에서 공통 split을 만들고, 세 시스템이 같은 test set을 예측한 뒤 결과를 비교표로 합칩니다.

설명 흐름: data -> systems -> evaluation -> comparison 순서로 말하면 전체 구조가 자연스럽게 이어집니다.

파일별 역할과 주요 함수

왼쪽 트리에서 파일을 선택하거나 위 실행 흐름의 항목을 선택하면 관련 파일 설명만 표시됩니다.

선택 대기 중 트리의 파일명 또는 실행 흐름 항목을 선택하세요.

src/system1_llm/predictor.py

file

검색 없이 물품 정보만으로 LLM이 10자리 HSK 코드를 예측합니다.

System 1은 외부 사례 검색 없이 LLM 자체 지식으로 HSK 코드를 직접 고르는 기준선입니다.
  • build_user_prompt()Record를 LLM 입력 프롬프트로 변환합니다.
  • predict()LLM 호출 결과를 Prediction 객체로 반환합니다.

entry

System 1 평가 진입점입니다.

이 파일은 System 1 predictor를 공통 runner에 연결하는 가장 얇은 실행 진입점입니다.
  • python -m src.system1_llm.run_evalLLM 단독 시스템을 평가합니다.

src/system2_rag/index.py

file

train split 텍스트를 embedding으로 바꾸고 FAISS 벡터 인덱스를 생성합니다.

이 파일은 train 데이터만 벡터화해서, 평가 시 유사한 과거 분류 사례를 검색할 수 있게 준비합니다.
  • get_embedder()sentence-transformers 모델을 필요할 때 한 번 로드합니다.
  • embed()텍스트 목록을 정규화된 float32 embedding으로 변환합니다.
  • NeighborMeta검색 결과의 문서 ID, HSK 코드, 물품 설명 metadata입니다.
  • build()FAISS index와 metadata cache를 생성합니다.
  • load()저장된 index를 읽거나 없으면 새로 만듭니다.
  • search()query text와 가까운 training 사례를 반환합니다.

src/system2_rag/predictor.py

file

RAG+LLM 예측과 순수 vector kNN baseline을 함께 제공합니다.

System 2는 유사 사례를 먼저 찾고, 그 근거를 LLM 판단이나 kNN 투표에 활용하는 검색 기반 방식입니다.
  • _format_neighbors()검색된 유사 사례를 LLM 프롬프트용 근거 텍스트로 만듭니다.
  • predict_knn()유사도 가중 투표로 코드 후보를 순위화합니다.
  • predict()유사 사례를 근거로 LLM이 최종 HSK 코드를 고르게 합니다.

src/system2_rag/run_eval.py

entry

System 2 평가 진입점입니다.

이 파일은 RAG+LLM 평가와 kNN baseline 평가를 명령 인자로 나누어 실행합니다.
  • python -m src.system2_rag.run_evalRAG + LLM 시스템을 평가합니다.

src/system3_graph/graph.py

file

term, case, code, HS hierarchy를 연결한 NetworkX 그래프를 만듭니다.

이 파일은 물품 단어, 과거 사례, HSK 코드, HS 계층을 하나의 그래프로 묶어 구조적 근거를 만듭니다.
  • tokenize()물품 텍스트를 검색용 token과 한국어 2-gram shingle로 분리합니다.
  • build()train records로 그래프와 IDF 정보를 생성하고 저장합니다.
  • load()저장된 graph cache를 읽거나 없으면 새로 만듭니다.

src/system3_graph/predictor.py

file

query term에서 case와 code로 activation을 전파해 후보 코드를 순위화합니다.

System 3은 LLM 없이 query 단어에서 관련 사례와 코드로 점수를 전파해 후보를 고르는 구조 기반 방식입니다.
  • _score_codes()term, case, hierarchy evidence를 합쳐 코드별 점수를 계산합니다.
  • predict()점수순 후보를 Prediction 객체로 반환합니다.

src/system3_graph/run_eval.py

entry

System 3 평가 진입점입니다.

이 파일은 그래프 cache를 준비한 뒤 System 3 predictor를 공통 평가 루프에 연결합니다.
  • python -m src.system3_graph.run_eval그래프 기반 시스템을 평가합니다.

file

경로, 모델명, RAG 검색 수, 평가 제한 등 전역 설정을 관리합니다.

이 파일은 실험에 필요한 경로, 모델, 평가 조건을 한 곳에서 조정하는 설정 중심입니다.

src/common/data.py

file

원천 JSONL을 읽고 모든 시스템이 공유하는 train/test split을 만듭니다.

이 파일은 원천 데이터를 읽고, 세 시스템이 같은 기준으로 평가되도록 공통 데이터 분할을 보장합니다.
  • Record물품명, 설명, 결정사유, 정답 HSK 코드를 담는 데이터 단위입니다.
  • load_records()유효한 10자리 단일 HSK 코드가 있는 레코드만 읽습니다.
  • split_records()코드별 빈도를 고려해 train/test를 안정적으로 분리합니다.
  • get_split()artifacts/split.json 캐시를 사용해 같은 split을 재사용합니다.

src/common/hscode.py

file

HSK 코드 정규화, 표시, 계층 비교, LLM 응답 파싱을 담당합니다.

이 파일은 HSK 코드를 내부 비교용 형식으로 맞추고, 예측 결과를 계층별로 비교할 수 있게 해줍니다.
  • normalize()코드에서 숫자만 남긴 내부 비교용 문자열을 만듭니다.
  • extract_code()자유 텍스트에서 첫 번째 10자리 HSK 코드를 추출합니다.
  • parse_llm_codes()LLM JSON 응답에서 대표 코드, 대안 코드, reasoning을 추출합니다.
  • level()chapter, heading, subheading, national 단위 코드를 반환합니다.
  • pretty()8536909010을 8536.90-9010 형식으로 바꿉니다.
  • match_level()예측 코드와 정답 코드가 특정 계층에서 일치하는지 비교합니다.

src/common/evaluation.py

file

예측 결과를 계층 정확도와 top-k 정확도로 평가하고 결과 파일을 저장합니다.

이 파일은 각 시스템의 예측 결과를 같은 지표로 계산해서 비교 가능한 report로 바꿉니다.
  • Prediction문서 ID, 정답, ranked 후보, latency, 부가 정보를 담습니다.
  • evaluate()chapter부터 national까지 계층별 top-1 정확도와 top-k exact를 계산합니다.
  • save_report()시스템별 report JSON과 prediction JSONL을 저장합니다.
  • print_report()콘솔에서 볼 수 있는 요약 평가 결과를 출력합니다.

src/common/runner.py

file

각 시스템의 run_eval.py가 공유하는 평가 실행 루프입니다.

이 파일은 test set을 순회하면서 predictor를 호출하고, 평가와 저장까지 이어주는 공통 실행 루프입니다.
  • run()test set 순회, predictor 호출, 예외 기록, 평가, 저장, 출력까지 수행합니다.

src/common/llm.py

file

Ollama chat API를 얇게 감싸 System 1과 RAG 시스템에서 재사용합니다.

이 파일은 Ollama 호출 방식을 한 곳에 모아 LLM을 쓰는 시스템들이 같은 인터페이스를 쓰게 합니다.
  • chat()system prompt와 user prompt를 보내고 assistant 응답 텍스트를 반환합니다.
  • ensure_model()지정한 Ollama 모델이 로컬에 있는지 확인합니다.

코드 설명 시 강조할 점

이 프로젝트의 핵심은 모델 자체보다, 세 시스템을 같은 조건에서 비교하도록 묶은 실험 구조입니다.

모든 시스템은 get_split()이 만든 같은 held-out test set을 사용합니다. RAG와 Graph는 train split만 인덱싱해야 하며, test label이 인덱스에 들어가면 비교 실험의 의미가 깨집니다.
